Runtime V2 keeps Taskplane’s durable execution model while replacing its fragile runtime control stack. The architecture explicitly preserves PROMPT.md, STATUS.md, .DONE, and .pi/batch-state.json as authoritative state, along with existing orchestration concepts like dependency DAGs, waves, lanes, and worktrees. What changes is the control path: TMUX-backed session orchestration and sidecar-based liveness are removed in favor of deterministic Node-owned process trees.

The target Runtime V2 structure is a strict ownership chain: extension command surface → engine process → lane-runner processes → agent-host processes → Pi RPC child processes. This design makes ownership unambiguous and reduces failure ambiguity, because each process has exactly one parent responsible for lifecycle and cleanup. Lane-runner is the per-lane execution authority for task progression, step state, .DONE handling, and snapshots, while agent-host handles RPC transport, telemetry normalization, mailbox steering integration, and event persistence.

Runtime V2 also improves observability and control without depending on terminal infrastructure. Steering is mailbox-first (supervisor to agent inbox, replies/escalations through outbox), and visibility comes from normalized structured events plus lane snapshots under .pi/runtime/{batchId}/. Agent IDs remain familiar (e.g., orch-*-lane-1-worker) but are now process-registry identities rather than TMUX session names. The intended end state is that /orch is the authoritative execution path even for one task, with correctness based on deterministic process ownership and recoverable file-backed state.
