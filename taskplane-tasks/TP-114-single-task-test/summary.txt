Runtime V2 is a redesign of Taskplane’s orchestration runtime that preserves durable execution artifacts (`PROMPT.md`, `STATUS.md`, `.DONE`, and `.pi/batch-state.json`) while replacing the fragile control-plane stack built around TMUX sessions and layered wrappers. The architecture diagnosis is that prior instability came from ambiguous process ownership, transport inversion (file polling for live state), and session lifecycle coupling. Runtime V2 keeps existing operator concepts—DAG planning, waves, lanes, supervisor, and dashboard—but moves execution onto deterministic Node-managed processes with explicit parent/child ownership.

The target architecture separates responsibilities across extension, engine, lane-runner, and agent-host components. The extension remains a command surface and supervisor entrypoint, the engine owns orchestration and policy decisions, each lane-runner owns serial task execution and checkpoint discipline in one lane, and each agent-host wraps a `pi --mode rpc` subprocess while handling telemetry normalization, mailbox steering, persistence, and lifecycle reporting. This model enforces a key rule: every spawned process has exactly one owner, so liveness and cleanup are deterministic and no longer inferred from terminal state.

Runtime V2 also standardizes observability and steering through runtime artifacts and mailbox flows instead of terminal capture. Agent events and manifests are persisted under `.pi/runtime/{batchId}/...`, lane snapshots are emitted directly, and the dashboard reads normalized logs for conversation and status views. The design explicitly removes correctness dependence on TMUX, makes packet-home path authority explicit for polyrepo execution, and positions `/orch` as the authoritative execution path (including single-task runs), with `/task` only a compatibility wrapper or eventual removal path after migration stability is achieved.
