Runtime V2 is Taskplane’s redesign of the runtime control plane: it keeps the durable execution model (`PROMPT.md`, `STATUS.md`, `.DONE`, and `.pi/batch-state.json`) but replaces TMUX-centered process orchestration with deterministic parent/child Node ownership. The architecture explicitly separates durable state (files) from live control/telemetry (direct process IPC), which reduces ambiguity about who owns each process and who must recover from failures.

The process model is hierarchical and explicit: extension → engine → lane-runner(s) → agent-host(s) → `pi --mode rpc`. Each layer has clear responsibilities: the extension handles command surface and supervisor UX, the engine handles orchestration policy and resumable batch state, lane-runners handle per-lane task execution loops and STATUS progression, and agent-hosts handle Pi subprocess lifecycle plus normalized event capture. This removes correctness dependencies on terminal infrastructure and avoids sibling processes inferring liveness from filesystem side effects.

Operationally, Runtime V2 emphasizes recoverability and observability through structured runtime artifacts under `.pi/runtime/{batchId}` plus mailbox-based steering. Agent identity stays familiar (e.g., `orch-...-lane-1-worker`) but is now process-registry-backed instead of terminal-session-backed. Success criteria focus on end-to-end stability: no TMUX dependency for worker/reviewer/merge lifecycle, dashboard visibility from normalized logs, mailbox-driven steering, explicit packet-path authority, and a unified `/orch`-first execution path even for single-task runs.
