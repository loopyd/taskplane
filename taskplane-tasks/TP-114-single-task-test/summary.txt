Runtime V2 is an architectural redesign of Taskplane's runtime control plane. While preserving the durable data plane — PROMPT.md, STATUS.md, .DONE files, batch-state.json, and the dependency DAG/waves/lanes model — it replaces the fragile TMUX-based execution infrastructure with a deterministic Node.js process hierarchy. The core problem it addresses is that the original runtime had too many overlapping layers (operator session → orchestrator extension → engine → TMUX lane session → RPC wrapper → Pi task-runner → TMUX worker session → RPC wrapper → Pi worker), leading to ambiguous process ownership, transport inversion where sibling processes communicated through file polling, and correctness dependencies on terminal infrastructure behavior.

The target architecture introduces a clean ownership chain: the extension owns the engine process, the engine owns lane-runners and merge hosts, each lane-runner owns its worker and reviewer agent-hosts, and each agent-host owns its Pi RPC child process. Two new components are central to this design: the lane-runner process, which replaces the pattern of running lanes as Pi sessions loading task-runner.ts and takes responsibility for serial task execution, STATUS parsing, step progression, and reviewer orchestration within a single lane; and the agent-host process, which serves as a transport adapter around "pi --mode rpc", handling subprocess lifecycle, live telemetry parsing, mailbox polling, steering injection, and event persistence. This ensures every spawned process has exactly one deterministic owner, eliminating the need to infer liveness from TMUX server state.

Runtime V2 also redefines agent identity from TMUX session names to process-registry-backed agent IDs, enables dashboard observability through structured event logs rather than pane capture, and establishes the mailbox as the sole steering path between operator, supervisor, and agents. The /orch command becomes the authoritative execution path even for single tasks, with legacy /task becoming a compatibility wrapper. Success is measured by the complete removal of TMUX dependencies for worker/reviewer/merge lifecycle, dashboard conversation rendering without pane capture, and explicit packet-path authority throughout segment execution.
