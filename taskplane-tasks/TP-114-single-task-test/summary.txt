Runtime V2 keeps Taskplane’s durable execution model while replacing the fragile runtime control plane. The architecture intentionally preserves `PROMPT.md`, `STATUS.md`, `.DONE`, `.pi/batch-state.json`, DAG/wave planning, and supervisor/dashboard concepts so recoverability and operator workflow remain stable. What changes is the control path: TMUX-backed orchestration and session-coupled identity are removed in favor of deterministic Node-owned child processes.

The target runtime is a clean ownership chain: extension command surface → engine process → lane-runner process per active lane → agent-host process per worker/reviewer/merge agent → `pi --mode rpc`. Each process has one deterministic owner, so lifecycle, failure attribution, and cleanup become explicit. Lane-runner owns per-lane task execution state, STATUS progression, and snapshot emission, while agent-host owns Pi subprocess transport concerns such as RPC parsing, mailbox steering injection, normalized event persistence, and exit summaries.

Runtime V2 also standardizes observability around structured runtime artifacts under `.pi/runtime/{batchId}` and mailbox/audit flows, rather than pane capture or sidecar polling as primary telemetry. Steering is mailbox-first (supervisor to agents, replies/escalations back through outbox), and packet-path authority is explicit to keep polyrepo correctness intact. The intended end state is that `/orch` is authoritative even for single-task execution, with `/task` reduced to compatibility behavior instead of being the mission-critical execution path.
