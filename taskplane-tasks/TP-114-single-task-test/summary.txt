Runtime V2 is Taskplane's architectural redesign that replaces the fragile TMUX-based runtime control plane with a deterministic, process-ownership-based execution model. The current system suffers from deeply nested process layers (operator session → extension → engine → TMUX lane → RPC wrapper → Pi task-runner → TMUX worker → RPC wrapper → Pi worker), which causes ambiguous ownership when processes fail, transport inversion where sibling processes communicate through file polling instead of direct IPC, and a correctness dependency on terminal infrastructure (TMUX) that was never designed for orchestration. Runtime V2 keeps Taskplane's proven durable data plane — PROMPT.md, STATUS.md, .DONE, batch-state.json, the dependency DAG, waves, lanes, worktrees, and orch branches — while replacing the control plane with a clean hierarchy where every spawned process has exactly one deterministic owner.

The new architecture introduces two key components: the lane-runner process and the agent-host process. The lane-runner replaces the pattern of running lanes as Pi sessions loading the task-runner extension; it owns exactly one lane worktree and handles serial task execution, STATUS parsing, step progression, checkpoint discipline, .DONE handling, and reviewer orchestration for that lane. The agent-host is a transport adapter around "pi --mode rpc" that owns the Pi subprocess lifecycle, live RPC event parsing, telemetry normalization, mailbox polling, conversation/event persistence, and exit summary generation. Together, these components create a simplified process tree: the extension owns the engine, the engine owns lane-runners and merge hosts, each lane-runner owns its worker/reviewer agent-hosts, and each agent-host owns its Pi child process.

Runtime V2 also transforms steering and visibility. Steering now flows exclusively through the mailbox system — the supervisor writes to agent inboxes, agent-hosts check the mailbox at message boundaries and inject messages via Pi RPC steer, and agents reply through their outbox. Visibility is achieved through structured event streams that agent-hosts persist directly, with the dashboard rendering full conversations from these logs rather than relying on TMUX pane capture. The specification introduces a batch-scoped runtime storage layout under .pi/runtime/{batchId}/ for agent manifests, event streams, lane snapshots, and a process registry. Importantly, the /task command path is deprecated in favor of /orch as the authoritative execution path even for single tasks, and features like persistent worker context, checkpoint discipline, quality gates, model fallback, and supervisor alert flows must all survive the redesign.
